# Skill Authoring (S2)

> Generated by init on 2026-04-11. Cite rules as S2.N in reviews.

## Anatomy Requirements

- **S2.1** Every SKILL.md must have YAML frontmatter with at minimum: `name`, `description`, `type: skill` (or `type: tech-stack` for tech stack skills).
- **S2.2** Required sections for workflow skills in order: `## Overview`, `## When To Use`, `## Workflow`, `## Verification`.
- **S2.3** Skill name in frontmatter must match the containing directory name exactly.
- **S2.4** Entry-point skills (user-invocable via `/mtk:<name>`) include `allowed-tools` and `argument-hint` in frontmatter. Workflow skills are reusable building blocks loaded by entry-point skills.

## Tech Stack Skills

- **S2.12** Tech stack skills (`tech-stack-{name}/SKILL.md`) provide declarative context, not workflow. They have `type: tech-stack` in frontmatter.
- **S2.13** Required sections for tech stack skills: `## Overview`, `## When To Use`, `## Build & Test Commands`, `## File Extensions & Markers`, `## ORM & Data Layer Guidance`, `## Framework Patterns`, `## Test Level Guidance`, `## Coding Style Reference`, `## Reference Files`, `## Settings Additions`, `## Format Command`, `## Scan Recipes`, `## Verification`. The `## Workflow` section is omitted (they're declarative, not procedural).
- **S2.14** Reference files for a tech stack live under `.claude/references/{stack}/`. The tech stack skill's `## Reference Files` section is the canonical list.
- **S2.15** When adding a new tech stack, the `## Coding Style Reference` may point to a placeholder file initially â€” write real guidelines when the team starts using that stack.

## CSO Principle

- **S2.5** Every `description` field (in manifest, plugin.json, and frontmatter) must be a single clear sentence explaining what the artifact does â€” not marketing copy, not a list.
- **S2.6** Descriptions should help Claude route to the right tool. "Reusable planning skill for substantial .NET changes" > "A skill for planning".

## Pressure Tests

- **S2.7** New skills that affect code review, security, or verification must have a pressure test in `tests/pressure-tests/`.
- **S2.8** Pressure tests present adversarial scenarios designed to make the skill skip steps or produce false positives.
- **S2.9** Follow `.claude/skills/writing-skills/SKILL.md` when creating or substantially modifying skills.

## Platform Frontmatter

- **S2.16** Use `effort: max` on review and security skills that must be thorough. Use `effort: high` on verification and drift-detection skills. Omit for lightweight skills.
- **S2.17** Use `context: fork` on skills that benefit from isolated context (review, security). This runs the skill in a separate subagent context.
- **S2.18** Use dynamic context injection (`` !`command` `` in fenced code blocks) to inject runtime state (tech stack, branch, diff stats) into skill content at load time. Prefer this over procedural "read file then decide" steps.

## Progressive Disclosure

- **S2.10** Skills should only load references they need for the current phase. Don't front-load all references.
- **S2.11** Commands that orchestrate multiple skills should specify which references each phase needs.
