# SQLAlchemy Checklist

Project-level reminders for reviewing and writing SQLAlchemy code (2.0 style).

## Query Rules

- Use the `select()` API (2.0 style), not the legacy `Query` API.
- For read-only queries, return DTOs via `Session.execute(select(...))` rather than full ORM objects when only projection is needed.
- Avoid implicit lazy loading in production code. Eager-load explicitly with `joinedload` or `selectinload`.
- Keep filtering in the database (`.where()` clauses), not in Python after `.all()`.
- Use `.scalars()` for single-column selects.

## Write Rules

- Wrap mutations in `with session.begin():` or use explicit `session.commit()` / `session.rollback()`.
- One commit per request/handler unless multiple aggregates require a multi-step transaction.
- Avoid mixing reads and writes in the same session without clear transactional boundaries.
- Detached objects don't track changes — be deliberate about session lifetime.

## Migrations

- Use Alembic. Don't bypass migrations with raw schema changes in code.
- Review autogenerated migrations before applying — Alembic guesses, especially for renames and constraint changes.
- One migration per logical change. Don't bundle unrelated schema changes.

## Test Provider

- SQLite in-memory works for pure logic tests but does NOT validate Postgres-specific behavior (JSONB, arrays, ILIKE, advisory locks, full-text search).
- For integration tests that depend on Postgres semantics, use `testcontainers-python` or a dedicated test database.

## Review Questions

- Is SQLAlchemy doing more work than necessary (loading too much, multiple round trips)?
- Is the query shaped correctly for the response?
- Will this behave correctly with the project's actual database (Postgres vs SQLite vs MySQL)?
- Are session and transaction boundaries explicit and minimal?
