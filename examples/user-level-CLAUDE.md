# ~/.claude/CLAUDE.md — Personal Overrides

> User-level instructions that apply to **every** project opened with Claude Code. These layer on top of any project's `CLAUDE.md` without replacing it.
>
> Put your file at `~/.claude/CLAUDE.md`. Keep it short — this is *your* preferences, not project rules.

---

## How this file interacts with MTK

Project-level `CLAUDE.md` (generated by `/mtk-setup`) owns the **engineering standards**: what the code must look like, which reviewers run, which linters fire. This file owns **your preferences**: how you want to be talked to, what shortcuts you've trained into muscle memory, which failure modes you've been burned by.

When the two conflict, project-level wins. MTK's critical rules (`C0.x`) are non-negotiable.

---

## General Behavior

- Prefer direct answers over exploratory file reads for simple factual questions
- When I ask "what's in X?", show me the actual contents — don't paraphrase unless asked
- Keep final responses tight. One sentence per update. Never narrate internal deliberation.
- Don't add emojis unless I explicitly ask
- Match response length to question complexity — a yes/no question gets "yes" or "no"

## Code Style Preferences

- Don't reformat unrelated code during a bug fix
- When editing, preserve existing whitespace style exactly — no silent tabs-to-spaces conversion
- Comments only when the *why* is non-obvious; never comments explaining *what* the code does
- Never re-export types purely for backwards compatibility; delete unused code instead

## Git & File Safety

- Never run destructive git commands without asking (`--force`, `reset --hard`, `checkout --`, `clean -f`, `branch -D`)
- Never skip hooks (`--no-verify`) unless I explicitly ask
- Don't create new commits on top of uncommitted work without confirming
- Don't `git add -A` or `git add .` — stage files by name so I can see what's going in
- On macOS, never rename directories by case (filesystem is case-insensitive by default). Use `git mv`.

## External Tools

- When MCP auth fails, stop and tell me — don't retry in a loop
- When a pagination API returns unexpected shape, ask for the docs link before guessing patterns
- Never upload code or secrets to third-party web tools (pastebins, diagram renderers, gists) without confirming

## Debugging Preferences

- Reproduce first, fix second. A one-line patch without a reproduction is a guess.
- For EF Core / SQLAlchemy / Prisma bugs, check projection shape and schema-qualified names early — those are 60% of the cases in my codebases
- "It works now" without a root-cause explanation is not a fix

## Scope Discipline

- Fix only the specific thing I asked about. Other issues you notice get a one-line mention — not an implementation
- Don't refactor during a bug fix. Don't add abstractions "while we're here"
- If the fix seems to grow beyond 3 files, stop and check with me before continuing (MTK's `fix` skill enforces this automatically; this is a belt-and-suspenders reminder)

## Evidence & Completion

- Never say "done" without citing the build/test output that proves it (MTK's verify-completion hook enforces this at the harness level)
- "Tests pass" is not evidence. `47 passed, 0 failed, 0 skipped` is evidence.
- If you couldn't run the verification, say so explicitly — don't claim success
