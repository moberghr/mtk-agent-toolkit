## Critical Rules (Always Apply)

### C1 — Type discipline
- **C1.1** No `any`. Use `unknown` at boundaries and narrow with type guards or Zod schemas.
- **C1.2** No `@ts-ignore`. Use `@ts-expect-error` with a comment citing the reason — the expect-error will break when the underlying issue is fixed, forcing cleanup.
- **C1.3** No non-null assertions (`!`) on API responses or user input. Narrow explicitly.
- **C1.4** Zod schemas own the runtime truth. DTOs are derived via `z.infer<typeof schema>`.

### C2 — Data & queries
- **C2.1** Prisma calls must use `select` or `include` explicitly — no "fetch full row, use one field" patterns (perf reviewer flags this).
- **C2.2** N+1 via Prisma relations: use `include` with nested filters, not `Promise.all` in a `.map()`.
- **C2.3** Money amounts are `Decimal` (Prisma `Decimal` type) — never `number`. Serialize as strings at the API boundary.
- **C2.4** No raw SQL with template literals. Use `Prisma.sql` tagged templates or `$queryRawUnsafe` with bound params.
- **C2.5** Transactions use `prisma.$transaction` — never manual begin/commit loops.

### C3 — Server vs client boundaries
- **C3.1** `"use server"` actions validate input with Zod before any DB access.
- **C3.2** Secrets (DB URL, auth secret, API keys) read from `env.ts` (t3-env) — never `process.env.X` scattered in code.
- **C3.3** `"use client"` components never import server-only modules (Prisma, auth session, server utils).
- **C3.4** Server components fetch data; client components render and interact. No fetching in client components except via TanStack Query.

### C4 — Auth & security
- **C4.1** Every tRPC procedure declares its auth requirement via middleware: `publicProcedure`, `protectedProcedure`, `adminProcedure`.
- **C4.2** Server actions wrap auth checks at the top — `const user = await requireUser()`. No "trust the client" patterns.
- **C4.3** CSRF: server actions and mutations require the built-in Next.js action encryption. Custom POST endpoints add explicit CSRF tokens.
- **C4.4** Raw HTML injection (e.g. `innerHTML`, React's unsafe-HTML escape hatch) requires a sanitizer pass and explicit reviewer sign-off. Review flags any new usage as critical.

### C5 — Testing
- **C5.1** Every tRPC procedure has a unit test for happy path + every Zod validation error.
- **C5.2** Components with conditional rendering have RTL tests for each branch.
- **C5.3** E2E tests cover: login, the primary create/read/update flow, and any payment or state-changing action.
- **C5.4** No `it.skip` / `describe.skip` without a linked issue. Slopwatch linter flags this.

### C6 — Architecture
- **C6.1** Feature-folder structure: `src/features/<feature>/` contains routers, components, schemas. Shared utilities in `src/lib/`.
- **C6.2** App Router layouts are thin. Business logic stays in server actions or tRPC procedures.
- **C6.3** No server-only logic in `middleware.ts` — that runs in the edge runtime and breaks DB clients.
