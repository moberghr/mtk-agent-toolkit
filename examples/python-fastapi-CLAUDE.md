## Critical Rules (Always Apply)

### C1 — Async & session discipline
- **C1.1** All I/O is async. Sync DB calls in an async handler are a bug, not a shortcut — the ruff config flags this.
- **C1.2** One request = one `AsyncSession`, injected via `Depends(get_session)`. Handlers never create sessions.
- **C1.3** Session lifetime is the request scope. Never pass sessions across async boundaries or store them in module globals.
- **C1.4** `asyncio.gather` for independent awaits; never chain `await` when operations are independent.
- **C1.5** No `asyncio.run` inside request handlers — the event loop already owns the request.

### C2 — Data & queries
- **C2.1** SQLAlchemy 2.0 style only: `select(User).where(...)`, not legacy `Query`. Pyright enforces via type stubs.
- **C2.2** No raw SQL with f-strings or `%` formatting. Use `text(":param")` with bound parameters.
- **C2.3** Money types use `Decimal` (Python stdlib) — never `float`. Pydantic schemas use `condecimal`.
- **C2.4** N+1 queries flagged by review: use `selectinload` or `joinedload` for known relationships; `selectin_polymorphic` for inheritance.
- **C2.5** Migrations are reviewed for safety: no `DROP COLUMN` without a deprecation window; no table locks on hot paths.

### C3 — Validation & security
- **C3.1** Every public endpoint has a Pydantic request model and a Pydantic response model. No `dict[str, Any]` in signatures.
- **C3.2** Auth is a FastAPI dependency — `Depends(require_user)` or `Depends(require_scope("..."))`. No inline auth checks.
- **C3.3** Secrets come from `Settings` (pydantic-settings) — never `os.environ.get` scattered in code.
- **C3.4** No PII in logs. Use structured logging with a redaction filter configured in `logging.yaml`.
- **C3.5** CORS is explicitly whitelisted — no `allow_origins=["*"]` in production profiles.

### C4 — Testing
- **C4.1** Every new endpoint has an integration test hitting FastAPI `TestClient` / `AsyncClient` with a real (Testcontainers) Postgres.
- **C4.2** Unit tests for pure logic; integration for DB / HTTP boundaries. Don't mock SQLAlchemy — use a real test DB.
- **C4.3** Coverage gate is 80% on `app/` modules. Below that, the CI pre-commit-review blocks.
- **C4.4** No `@pytest.mark.skip` without a linked issue. Slopwatch linter flags this.

### C5 — Architecture
- **C5.1** Layers: `api/` (routers) → `services/` (business logic) → `repositories/` (data access) → `models/` (SQLAlchemy) + `schemas/` (Pydantic). Crossing layers bottom-up is a review blocker.
- **C5.2** Dependency injection via `Depends` — no manual wiring, no globals.
- **C5.3** Background work goes to a task queue (Celery/Arq), not `BackgroundTasks` for anything that must survive a restart.
